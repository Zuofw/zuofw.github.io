下游数据上报时，在多个上报中，如何处理数据一致性

  1. 可以设置版本号或者唯一标识来保证数据一致性。
  2. 事务管理，将多个相关数据上报操作封装在一个事务中，如果在事务处理过程中出现任何错误，事务可以回滚，保证数据一致性。

Redis分布式锁的缺陷和解决方案

  1. 锁失效：如果业务执行时间超过锁过期时间，可能会导致锁被其他线程获取，从而导致数据不一致。
    • 解决方案就是使用看门狗机制和设置超时时间
  2. 锁误删问题：线程A释放锁时可能误删线程B的锁(如A的锁过期后，B获取信的锁，但是A在执行解锁操作)
    • 解决方案是：使用唯一标识或者使用Lua脚本来实现校验标识+删除锁的原子操作，避免中间状态。
  3. 主从同步延迟问题：主节点写入锁后位同步到从节点即宕机，可能导致其他线程从新主节点获取重复锁。
    • 解决方案：RedLock算法、使用ZooKeeper来保证强一致性

ForkJoin的核心原理

核心思想是分治算法和工作窃取机制，适用于大规模并行任务的高效处理。
- 分支算法：将大任务拆分为子任务(fork)，直到子任务足够小，直接计算结果后逐层合并(join)
- 工作窃取机制：
- 每个工作线程维护一个双端队列，优先从队尾取出自己的任务执行
- 当线程空闲时，从其他线程的队头窃取任务，减少线程阻塞。
- 数据结构使用的是ForkJoinTask，任务队列基于WorkQueue优化

手撕生产者消费者模式(多线程并发版，synchronized的使用方式)

public class ProducerConsumerWithBoundedArray {  
    private final int[] buffer;  
    private int count = 0; // 当前元素数量  
    private int putIndex = 0; // 下一个插入位置  
    private int takeIndex = 0; // 下一个取出位置  
  
    public ProducerConsumerWithBoundedArray(int capacity) {  
        buffer = new int[capacity];  
    }  
  
    // 生产者  
    class Producer implements Runnable {  
        @Override  
        public void run() {  
            while (true) {  
                synchronized (this) {  
                    while (count == buffer.length)  { // 缓冲区满则等待  
                        try {  
                            wait();  
                        } catch (InterruptedException e) {  
                            e.printStackTrace();  
                        }  
                    }  
                    int item = (int) (Math.random()  * 100);  
                    buffer[putIndex] = item;  
                    putIndex = (putIndex + 1) % buffer.length;  // 循环数组  
                    count++;  
                    System.out.println(" 生产: " + item + "，当前数量: " + count);  
                    notifyAll(); // 唤醒消费者  
                }  
                try {  
                    Thread.sleep(500);  // 模拟生产耗时  
                } catch (InterruptedException e) {  
                    e.printStackTrace();  
                }  
            }  
        }  
    }  
  
    // 消费者  
    class Consumer implements Runnable {  
        @Override  
        public void run() {  
            while (true) {  
                synchronized (this) {  
                    while (count == 0) { // 缓冲区空则等待  
                        try {  
                            wait();  
                        } catch (InterruptedException e) {  
                            e.printStackTrace();  
                        }  
                    }  
                    int item = buffer[takeIndex];  
                    takeIndex = (takeIndex + 1) % buffer.length;  // 循环数组  
                    count--;  
                    System.out.println(" 消费: " + item + "，剩余数量: " + count);  
                    notifyAll(); // 唤醒生产者  
                }  
                try {  
                    Thread.sleep(1000);  // 模拟消费耗时  
                } catch (InterruptedException e) {  
                    e.printStackTrace();  
                }  
            }  
        }  
    }  
  
    public static void main(String[] args) {  
        ProducerConsumerWithBoundedArray pc = new ProducerConsumerWithBoundedArray(5);  
        new Thread(pc.new  Producer()).start();  
        new Thread(pc.new  Consumer()).start();  
    }  
}

一致性哈希详解

原理

将节点(物理或者虚拟)和数据通过哈希函数映射到固定范围内的原型空间。可以根据节点IP或者唯一标识计算哈希值。数据哈希值按照顺时针方向找到最近的节点作为目标节点。 可以给每个物理节点对应多个虚拟节点，通过哈希分散到环的不同位置，解决数据倾斜问题。

节点故障处理和替代节点选择

替代节点选择：当节点宕机时，原本路由到该节点的数据会顺时针找到环的下一个节点作为新归属。
数据迁移范围：仅影响宕机节点到其前驱节点之间的数据，其他节点不受影响。

重新哈希确定位置

新增节点时，通过哈希函数计算其IP或者标识的哈希值，确定在哈希环上的位置。
通过虚拟节点机制，新增节点的虚拟位置会均匀分布在环上，减少数据迁移量。

如何处理高QPS节点

  1. 动态负载均衡：
    • 虚拟节点扩容：为更高负载节点动态增加虚拟节点的数据量
    • 对热点数据生成多个副本，映射到哈希环上的不同位置
  2. 本地缓存与请求限流：
    • 客户端或者代理层对热点数据增加本地缓存，减少对节点的直接访问
    • 通过令牌桶等算法限制单个节点的请求速率。

大量节点宕机后的虚拟节点拆分

虚拟节点重新分配：
按照权重分配：根据剩余节点的容量比例分配虚拟节点
哈希均匀性：通过哈希函数将原虚拟节点重新映射到剩余节点。new_virtual_node = hash(old_virtual_node + "_replica") % 2^32 

import java.util.*; 
import java.security.MessageDigest; 
 
public class ConsistentHash {
    // 哈希环存储虚拟节点（TreeMap实现有序环结构）
    private final TreeMap<Long, String> virtualNodes = new TreeMap<>();
    private final int virtualNodeCount; // 每个物理节点对应虚拟节点数 
    private final List<String> downNodes = new ArrayList<>(); // 宕机节点记录 
 
    public ConsistentHash(int virtualNodeCount) {
        this.virtualNodeCount  = virtualNodeCount;
    }
 
    // 哈希生成算法（基于MD5）
    private long hash(String key) {
        try {
            MessageDigest md5 = MessageDigest.getInstance("MD5"); 
            byte[] digest = md5.digest(key.getBytes()); 
            return ((digest[3] & 0xFFL) << 24) | ((digest[2] & 0xFFL) << 16)
                    | ((digest[1] & 0xFFL) << 8) | (digest[0] & 0xFFL);
        } catch (Exception e) {
            throw new RuntimeException("MD5 not supported");
        }
    }
 
    // 添加物理节点（自动创建虚拟节点）
    public void addNode(String node) {
        for (int i = 0; i < virtualNodeCount; i++) {
            String virtualNode = node + "#VN" + i;
            long hash = hash(virtualNode);
            virtualNodes.put(hash,  node);
        }
    }
 
    // 移除故障节点（自动移除所有虚拟节点）
    public void removeNode(String node) {
        Iterator<Map.Entry<Long, String>> it = virtualNodes.entrySet().iterator(); 
        while (it.hasNext())  {
            Map.Entry<Long, String> entry = it.next(); 
            if (entry.getValue().equals(node))  {
                it.remove(); 
            }
        }
        downNodes.add(node);  // 记录宕机节点 
    }
 
    // 请求路由（带故障转移机制）
    public String route(String key) {
        if (virtualNodes.isEmpty())  throw new RuntimeException("No available nodes");
        
        long hash = hash(key);
        Map.Entry<Long, String> entry = virtualNodes.ceilingEntry(hash); 
        if (entry == null) { // 环状处理：返回第一个节点 
            entry = virtualNodes.firstEntry(); 
        }
        
        // 检查节点健康状态（此处简化实现）
        if (downNodes.contains(entry.getValue()))  {
            return findNextAvailableNode(hash);
        }
        return entry.getValue(); 
    }
 
    // 故障节点自动跳过逻辑 
    private String findNextAvailableNode(long hash) {
        SortedMap<Long, String> tailMap = virtualNodes.tailMap(hash); 
        for (Map.Entry<Long, String> entry : tailMap.entrySet())  {
            if (!downNodes.contains(entry.getValue()))  {
                return entry.getValue(); 
            }
        }
        return virtualNodes.firstEntry().getValue(); 
    }
}

CompletableFuture和FutureTask源码相关

看我自己的源码分析，这里主要记录获取结果的实现方式

FutureTask

使用volatile int state标识任务状态(NEW、COMPLETING、CANCELLED)，通过AQS的变种来实现线程阻塞和唤醒。

public V get() throws InterruptedException, ExecutionException {
    int s = state;
    if (s <= COMPLETING) // 未完成时阻塞 
        s = awaitDone(false, 0L);
    return report(s); // 返回结果或异常 
}

get()方法会使得调用线程被挂起，直到任务完成或者超时才返回结果。

CompletableFuture

使用无锁的CAS操作和原子变量来管理状态，每个阶段通过链表结构记录依赖关系，避免锁竞争。
支持异步回调获得结果，通过注册回调函数，任务完成后自动触发回调，主线程无需阻塞。

  1. 链式调用实习原理：实现了CompletionStage接口，这个接口定义了分阶段任务编排的能力，通过任务执行后返回一个新的CompletionStage对象，形成链式调用
  2. 任务依赖管理：每个阶段任务通过栈存储依赖关系。当一个异步任务完成时，会从栈顶依次触发后续的回调任务，确保执行顺序可控
    回调节点Completion的设计，通过继承ForkJoinTask并实现Runnable代表一个异步操作节点。
    abstract static class Completion extends ForkJoinTask<Void> 
    implements Runnable, AsynchronousCompletionTask {
    volatile Completion next;  // 指向下一个节点，构成链表 
    // 核心方法：触发任务执行 
    abstract void run();
    abstract void tryFire(int mode);
}
    操作压栈：
    public <U> CompletableFuture<U> thenApply(Function<? super T,? extends U> fn) {
    return uniApplyStage(null, fn);
}
 
private <V> CompletableFuture<V> uniApplyStage(
    Executor e, Function<? super T,? extends V> f) {
    CompletableFuture<V> d = new CompletableFuture<V>();
    if (e != null || !d.uniApply(this,  f, null)) {
        UniApply<T,V> c = new UniApply<T,V>(e, d, this, f);
        push(c);  // 压栈操作 
        c.tryFire(SYNC); 
    }
    return d;
}
 
// 压栈逻辑（通过 CAS 操作保证线程安全）
final void push(Completion c) {
    do {} while (!tryPushStack(c));
}
    回调触发：当调用complete()或者返回结果时，会遍历栈中的Completion节点并且触发后续的操作。
    final void postComplete() {
    CompletableFuture<?> f = this;
    Completion h;
    while ((h = f.stack)  != null) {
        CompletableFuture<?> d; Completion t;
        if (f.casStack(h,  t = h.next))  {  // 出栈操作 
            if (t != null) {
                if (f != this) {
                    pushStack(h);
                    continue;
                }
                h.next  = null;  // 断开链表 
            }
            f = (d = h.tryFire(NESTED))  == null ? this : d;
        }
    }

本地方法栈是什么

本地方法栈是JVM内存结构的一部分，专门用于管理本地方法的调用和执行，每个线程独立拥有属于自己的本地方法栈，生命周期和线程绑定。

本地方法在本地方法栈中的存储方式是什么

栈帧结构：
本地方法调用时，本地方法栈会为该方法分配一个战阵，包含以下内容：

  1. 局部变量表：存储方法的参数和局部变量
  2. 操作数栈：用于执行本地方法的指令操作
  3. 动态连接：指向方法所属类的运行时常量池引用
  4. 方法出口：记录方法执行完成后的返回地址。
    本地方法的实际执行由操作系统完，JVM仅通过本地方法栈管理其调用状态，调用本地方法时，JVM不会在Java虚拟机中创建栈帧而是直接操作本地方法栈。

什么是方法内联

方法内联是JVM的编译器优化技术，将方法调用替换成方法体的实际代码，以减少函数调用的开销(栈帧创建、上下文切换)
原理：
当某个方法(如短小的Getter/Setter)被频繁调用时，JIT编译器会将其代码直接嵌入调用出，避免跳转开销。

本地方法可以内联吗

本地方法通常不能够内联，因为其实现在JVM外部，编译器无法访问其代码逻辑
内联优化主要针对Java方法，本地方法需要通过JNI接口调用，存在额外性能损失。
